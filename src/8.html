배운거 총연습 part 2. 코드 혼자 짜보기-영화 평점 리스트 (Home.js 참조) 1. 에러정리 - 자바스크립트 배열 []을 JSX에서 {test}로 써버리면 "Objects are not valid as a React child" 콘솔 에러가 뜬다 구글 검색에서는 해당 에러는 타입이 안 맞을 때 뜬다고도 한다. JSX에선 Object를 그대로 뿌려 줄 수 없다는 뜻인 거 같다. 2. 새로운 방식의 Ajax 티스토리에 적어 놓음 - https://stir.tistory.com/92 3. 라우터 사용법 - npm i react-router-dom@5.3.0로 라우터 설치 / App.js에 라우터를 설정후에 각각의 페이지 이동은 로 연결한다 {title} 라고 쓸 수 있는데 여기서 템플릿 리터럴이 쓰인다. 템플릿 리터럴은 ES6부터 나왔다. 템플릿 리터럴은 ${}로 표현식을 사용한다. 문자를 +로 연결하지 않아도 된다. ${id}로 사용 받은 값은 useParams를 import해서 문법대로 구현하면 라우터에서 받아온 값을 사용 가능하다. 4. 클라이언트 사이드 렌더링(CSR) vs 서버 사이드 렌더링(SSR) 영화 평점 리스트 화면을 만들면서 페이지 이동을 할 수 있는 Router라는 것을 쓰게 되는데, 페이지 이동을 한다고 해서 JSP와 기능이 같은 것은 아니다. 왜냐면 React는 CSR 방식을 쓰기 때문이다. SSR은 기존에 JSP/Servlet의 아키텍처에서 이 방식을 사용했다. 페이지를 이동할 때마다 새로운 페이지를 전체 요청 한다. CSR은 클라이언트에서 페이지를 가져와서 보통의 Vue나 React가 기본적으로 사용하는 방식이다. SPA(Single Page Application)을 두고 부르는 말 이기도 하다. 다만 CSR은 SEO(검색 엔진 최적화)에는 최적화 되어있지 않다. CSR은 기본적으로 초반에 빈 html을 제공 후에 채워넣기 때문에 검색 엔진은 이를 빈 페이지라고 받아들인다. 속도는 빠르지만 초기 로딩에 모든 자바스크립트를 가져와야해서 굉장히 많은 시간이 소요된다. 어떠한 어플리케이션을 만드느냐에 따라 둘은 장단점이 존재한다. Vue, React는 검색엔진 최적화 문제를 해결하기 위해 Next.js 프레임워크를 사용한다. Next.js는 첫페이지는 백엔드 서버에서 렌더링하여 빈 html이 아닌 데이터가 채워진 html을 받아 검색 최적화 문제를 해결하고 그 다음 페이지부턴 CSR방식을 적용하여 필요한 데이터 부분만 갱신해 서버의 부하도 줄이게 한 것이다. 즉 첫페이지의 느림과 검색엔진최적화 문제를 해결한 프레임워크다. Next.js의 또 다른 장점은 라우팅이 매우 간단한다. pages라는 폴더 안에 넣은 파일명이 바로 라우팅이 되는 방식이다. 예를 들어 mypage.js라는 파일을 만들면, localhost:3000/mypage으로 자동으로 라우팅이 된다.
